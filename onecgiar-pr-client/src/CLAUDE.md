# CLAUDE.md — `onecgiar-pr-client/src` (source-tree navigation & patterns)

This is the **source-tree guide** for the PRMS Angular client. It complements the package-level [`../CLAUDE.md`](../CLAUDE.md), the root [`../../CLAUDE.md`](../../CLAUDE.md), and the SDD baseline under [`../../docs/`](../../docs/).

> **Read order:** root → package → this file. Everything below assumes you already know the SDD methodology and the high-level architecture (Angular 19 + PrimeNG 19, custom `auth` header, no NgRx).

This file is intentionally dense. It documents:

- Every top-level folder/file under `src/`, what lives in it, and why.
- The exact patterns to follow when adding a new page / component / service.
- The patterns to **stop and ask before changing** (load-bearing infrastructure).
- Naming conventions and anti-patterns observed in the current tree.

## 1. Bootstrap & wiring (root files)

### `main.ts`

Trivial Angular bootstrap: `enableProdMode()` in prod, `platformBrowserDynamic().bootstrapModule(AppModule)`. **Do not** add side effects here — global one-time setup belongs in `app.module.ts` via `provideAppInitializer`.

### `polyfills.ts`

`zone.js` only. No IE legacy polyfills. Don't bloat.

### `index.html`

PRMS SPA shell. Contains the `<app-root>` outlet and meta tags. Avoid editing unless you're adding analytics SDKs (Tawk, Hotjar, Clarity) — most of these are injected from components instead.

### `styles.scss`

Global stylesheet entry. Loads `./styles/fonts.scss` and `primeicons.css`, then defines project-wide utility classes:

- **Custom scrollbar** `.custom_scroll` — opt-in via class, themed from `--pr-color-primary-*`.
- **Status / completeness pills** `.completeness-submitted | .completeness-editing | .completeness-all | .completeness-quality-assessed | .completeness-discontinued` and `.result-list-state` — use these for `status_id` chips so the workflow color mapping stays consistent across screens.
- **`.in-qa-tag`** — small "in QA" badge.
- **Margin utilities** generated by `@include generate-margin-right-classes(100)` — produces `.m-N`, `.mr-N`, `.ml-N`, `.mt-N`, `.mb-N`, `.mx-N`, `.my-N` for `N = 1..100`. Available globally; prefer these over inline `style="margin: ..."`.
- **`.globalDisabled`** — visual-only disable (`opacity: 0.5; pointer-events: none`). Always pair with the actual disabled attribute on the form control.
- **P25 drawer scroll lock** `body.pr-p25-drawer-scroll-lock { ... }` — used by the P25 column drawer to keep the sticky header working when scroll is locked. Don't repurpose this class.

Don't add new ad-hoc utility families here — prefer `src/styles/<topic>.scss` and import where needed.

### `setup-jest.ts`

`setupZoneTestEnv()` from `jest-preset-angular`. That's it.

## 2. `app/` — Angular application

### 2.1 `app.module.ts`

Single source of truth for shell-level providers and imports. Key things wired here:

| Topic | What's set | Notes |
|---|---|---|
| Routing | `AppRoutingModule` | Composed from `extraRoutingApp + routingApp`. |
| HTTP | `provideHttpClient(withInterceptorsFromDi())` + `{ provide: HTTP_INTERCEPTORS, useClass: GeneralInterceptorService, multi: true }` | The interceptor attaches the custom `auth` header. |
| PrimeNG theme | `providePrimeNG({ theme: { preset: reportingTheme, options: { darkModeSelector: 'light' } } })` | **`darkModeSelector: 'light'`** forces light. Dark mode is not supported. |
| Animations | `provideAnimationsAsync()` | Required by PrimeNG. |
| Toast | `MessageService` + `ToastModule` | Global toast via `messageService.add({ key: 'globalUserNotification', ... })`. |
| Clarity | `provideAppInitializer(() => initializeClarityService(inject(ClarityService)))` | Microsoft Clarity bootstraps once on app init. |
| Sockets | `SocketIoModule.forRoot(...)` + `WebsocketService` are **commented out** | The WebSocket layer is dormant. Re-enabling it requires coordination — don't flip it on silently. |
| Modals registered as imports | `ShareRequestModalModule`, `ChangePhaseModalModule`, `OverlayBadgeModule`, `BadgeModule`, `DialogModule`, `TooltipModule`, `SatPopoverModule`, ... | Shell-level components and modals are pre-imported so any page can trigger them. |
| Standalone components imported here | `HeaderPanelComponent`, `NavigationBarComponent`, `TawkComponent`, `PopUpNotificationItemComponent`, `FormatTimeAgoPipe`, `UserRolesInfoModalComponent` | The shell chrome lives at this level. |

**Rule:** don't add app-wide providers without a spec. New page-specific providers belong in the feature module, not here.

### 2.2 `app-routing.module.ts`

A one-liner: `RouterModule.forRoot([...extraRoutingApp, ...routingApp])` (see §4 for the route tables).

### 2.3 `app.component.{ts,html,scss}`

The app shell. Hosts the header, navigation, footer, toast outlet, global modals (share request, change phase), and the `<router-outlet>`. Don't put feature logic here — extract into `shared/components/` and import.

## 3. `app/pages/` — Top-level surfaces

13 folders, one per route group. **Every feature follows the same shape.**

### 3.1 Standard feature module layout

```
pages/<feature>/
├── <feature>.module.ts
├── <feature>-routing.module.ts
├── <feature>.component.{ts,html,scss,spec.ts}
├── <feature>.service.ts            # Optional, feature-level state/state-orchestration
├── (optional) <feature>.responsive.scss  # Feature-level responsive overrides
├── components/                     # Feature-only components (subfolder per component)
├── pages/                          # Nested routed sub-pages (mirrors this same shape)
└── services/                       # Feature-local services
```

Each `components/<name>/` and nested `pages/<sub-page>/` follows the same 4-file Angular convention:

```
<name>/
├── <name>.component.html
├── <name>.component.scss
├── <name>.component.spec.ts
└── <name>.component.ts
```

Modules (`.module.ts`) are excluded from coverage. Routing modules (`*-routing.module.ts`) are excluded from coverage.

### 3.4 Adding a new feature module under `pages/`

1. Create `pages/<feature>/` with the standard layout (§3.1).
2. Register the route in `app/shared/routing/routing-data.ts`:
   - Top-level → add to `routingApp` (or `extraRoutingApp` if admin-gated).
   - Sub-route → add to the appropriate nested table (`resultRouting`, `adminModuleRouting`, `notificationsRouting`, etc.).
   - Use `PrRoute` (the project's `Route` extension) so `prName`, `prHide`, `underConstruction`, `onlyTest`, `portfolioAcronym` are available.
3. Apply guards: `CheckLoginGuard` for authenticated routes, `CheckAdminGuard` for admin-only.
4. Lazy-load via `loadChildren` (preferred for full feature modules) or `loadComponent` (for single-page features).
5. Add at least the four PrimeNG components you need to the new `.module.ts`. Don't import all of PrimeNG.
6. If the feature has multi-section editing, model it as a `panel-menu` + nested `pages/`. Don't reinvent navigation.

## 4. `app/shared/routing/`

Single file: `routing-data.ts`. Exports many `PrRoute[]` tables consumed by feature `*-routing.module.ts` files.

### 4.1 `PrRoute` extension

```ts
export interface PrRoute extends Route {
  prName: string;                 // Display label for nav / breadcrumbs / panel-menu
  prHide?: boolean | number;      // true → hide; number → tie route to a result_type_id
  underConstruction?: boolean | number;
  onlyTest?: boolean;             // Limit to test env
  portfolioAcronym?: string;      // 'P22' | 'P25' — controls portfolio-version visibility
}
```

The number form of `prHide` is **load-bearing for result-detail navigation** (it maps to `ResultTypeEnum`). Don't strip it.

### 4.2 Route tables to know

| Constant | Used by | Purpose |
|---|---|---|

Fallback redirect → `result-framework-reporting`. |

### 5.4 Service rules

- **One service per concern.** If the file is creeping past ~400 lines, split it.
- **State → signals first, RxJS where needed.** Components SHOULD bind to `*Signal` properties; legacy `BehaviorSubject` fields are being migrated.
- **Service methods are idempotent unless they explicitly mutate.** Pure reads should not push to UI state as a side effect.
- **`HTTP_METHOD_descriptiveName`** naming — apply to every HTTP method you add.

## 6. `app/shared/interceptors/` — `GeneralInterceptorService`

`general-interceptor.service.ts` is the project's only HTTP interceptor. Its responsibilities:

1. **Attach `auth` header** with `localStorageToken` to every request — except requests whose URL contains `environment.elastic.baseUrl` (the client hits Elasticsearch directly with separate credentials).
2. **Side effects on success of `PATCH` / `POST`:**
   - If the current router URL contains `/result/result-detail/` → call `GreenChecksService.getGreenChecks()`.
   - If the request URL contains `/api/ipsr/` AND is not in the small denylist (`['/api/ipsr/all-innovations']`) → call `IpsrCompletenessStatusService.updateGreenChecks()`.
3. **Errors** bubble through `manageError` for caller-side handling. No automatic toast — components / services are responsible for surfacing errors.

**Rule:** If a new flow shouldn't trigger the completeness refresh, add its URL to the `notValidateList` inside the interceptor — **do not bypass the interceptor**.

## 7. `app/shared/guards/`

| Guard | Purpose | Used by |
|---|---|---|
| `CheckLoginGuard` | If `authService.localStorageToken` exists → allow; else redirect to `/login`. | `routingApp` (most top-level routes). |
| `CheckAdminGuard` | Async — calls `GET_allRolesByUser()` and checks `response.application.role_id === 1`. On failure redirects to `/result/results-outlet/results-list`. | `extraRoutingApp` (admin module). |

**Rules:**

- Admin sections MUST be gated by `CheckAdminGuard`. Hiding nav items is not enough (AC-3 in `../../docs/prd.md`).
- Don't add a new guard unless the role model demands it — extend `CheckAdminGuard` patterns first.

1. `app/shared/components/<name>/` with the 4-file layout.

## 11. `app/internationalization/` — Terminology service

The PRMS i18n is **portfolio-driven**, not locale-driven:

```ts
@Injectable({ providedIn: 'root' })
export class TerminologyService {
  t(key: TermKey, portfolioAcronym: string): string {
    const dict = portfolioAcronym === 'P25' ? NEW_TERMS : LEGACY_TERMS;
    return dict[key] ?? key;
  }
}
```

Dictionaries `LEGACY_TERMS` and `NEW_TERMS` are defined in `terminology.config.ts`. Static seed data lives in `app/shared/data/internationalization-data.ts`.

### Rules

- All user-facing copy that differs between **P22 and P25** MUST be authored as a `TermKey` and consumed via the pipe `term`.
- Locale (English / Spanish / etc.) is **not** in scope today — the keys are ready for future locale extension but the service only switches portfolios.
- Hardcoded English IS acceptable for purely structural / non-domain copy (`"Cancel"`, `"OK"`) — but consider promoting to `TermKey` if it's recurring.

## 21. Patterns to follow

### 21.1 Service shape

```ts
@Injectable({ providedIn: 'root' })
export class FooService {
  private readonly api = inject(ResultsApiService);
  readonly fooSignal: WritableSignal<Foo | null> = signal(null);

  loadFoo(id: number) {
    this.api.GET_fooById(id).subscribe(({ response }) => this.fooSignal.set(response));
  }
}
```

- `@Injectable({ providedIn: 'root' })` for shared services.
- Prefer `inject()` over constructor params for new code; preserve existing constructor injection where present.
- State on `WritableSignal<T>` for new code.
- Destructure `response` from the standard envelope.

### 21.2 Component shape

```ts
@Component({
  selector: 'app-foo',
  templateUrl: './foo.component.html',
  styleUrls: ['./foo.component.scss']
})
export class FooComponent {
  private readonly api = inject(ApiService);
  readonly fooSignal = computed(() => this.api.dataControlSE.currentResultSignal());
}
```

- Reach feature services through `ApiService` (the aggregator) — `this.api.dataControlSE`, `this.api.resultsSE`, etc.
- Templates bind to signals: `{{ fooSignal() }}`.
- Forms via PrimeNG `pr-input` / `pr-select` (from `custom-fields/`).
- No inline `style="..."`. Use `--pr-*` classes or component SCSS.

### 21.3 HTTP service methods

```ts
GET_resultById(id: number) {
  return this.http.get<ReportingApiResponse<CurrentResult>>(`${environment.apiBaseUrl}api/results/${id}`);
}
```

- Name as `HTTP_METHOD_descriptiveName`.
- Use the right base URL (see [`../CLAUDE.md`](../CLAUDE.md) §2). The interceptor will attach the `auth` header.
- Type the response. Don't return `any`.

### 21.4 Side effects

- The interceptor handles green-checks and IPSR completeness refresh post-PATCH/POST. Add new exclusions to `notValidateList` inside the interceptor; don't bypass it.
- Real-time events (sockets, Pusher) are **hints** — always reconcile via API before mutating UI state.

### 21.5 Forms & validation

- Use `feedback-validation.directive.ts` for server-side feedback fields.
- Programmatic labels: every form control has a `for=` label or an `aria-label`.
- Required-section indicators surface through `green_checks` in `DataControlService`.

### 21.6 i18n

- `term` pipe (`internationalization/`) for any copy that differs P22 vs P25.
- Keep dictionaries up to date in `terminology.config.ts`.

### 21.7 Modals

- Reuse `share-request-modal`, `change-phase-modal`, etc. Don't reinvent.
- New global modals → `shared/modals/<name>/`, imported in `app.module.ts`, triggered via a flag on `DataControlService`.

### 21.8 Telemetry

- Clarity initialises in `app.module.ts` via `provideAppInitializer`. Mocked in `tests/mocks/clarityMock.ts`.
- Hotjar via environment ids.
- **NEVER log tokens, the full `auth` header, webhook URLs, or AD credentials** (`../../.cursorrules`).

## 22. Anti-patterns to avoid

- **Hardcoded English in templates** — use the `term` pipe and dictionaries.
- **Inline `style="margin: 12px"`** — use `.m-12`, `.mb-12`, etc. from `styles.scss`.
- **Hex literals in SCSS** — extend `colors.scss` first, then consume `var(--pr-color-*)`.
- **Bypassing `GeneralInterceptorService`** — even Elasticsearch is opt-out *inside* the interceptor.
- **Per-feature soft-delete / green-checks logic** — centralize in `data-control.service.ts` and `green-checks.service.ts`.
- **Renaming the misspellings** (`steper-navigation`, `TypePneReportRouting`, `managementPhasesRuting`, `chagePhaseModal`, `typeOneResportEmbedded`) — they're in code paths and config keys consumed elsewhere.
- **Adding new top-level folders under `app/`** — extend `shared/` instead.
- **Importing all of PrimeNG in `app.module.ts`** — per-feature module imports keep the bundle reasonable.
- **Re-enabling `WebsocketService` silently** — it's commented out for a reason; coordinate with the server team.
- **Hiding admin UI without a guard** — `CheckAdminGuard` is the enforcement point.
- **Pushing production secrets into `environment.prod.ts`** — keys here are checked in.
- **Bypassing `ResponseInterceptor` envelope** — destructure `response`, don't read the whole body.
